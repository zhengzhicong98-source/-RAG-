# Overly Permissive RLS Policy - quality_scores INSERT Allows Anonymous Writes

**ID:** vuln-0002
**Severity:** MEDIUM
**Found:** 2026-07-12 09:58:09 UTC
**Target:** https://github.com/zhengzhicong98-source/legal-assistant
**Endpoint:** supabase/migrations/00028_add_quality_scores.sql
**CWE:** CWE-862
**CVSS:** 5.3

## Description

The `quality_scores` table in migration 00028 has an INSERT RLS policy that allows any request (including unauthenticated/anonymous requests) to insert rows. The policy `"insert scores" ON quality_scores FOR INSERT WITH CHECK (true)` has no authentication or authorization requirements.

## Impact

An unauthenticated attacker can arbitrarily insert fraudulent quality scoring data into the quality_scores table. This can pollute the quality statistics view (quality_stats) that powers the admin dashboard, causing incorrect AI evaluation metrics. In a production context, this could mask actual quality issues or generate false alerts.

## Technical Analysis

Migration `supabase/migrations/00028_add_quality_scores.sql` at line 34 creates an RLS policy:

```sql
CREATE POLICY "insert scores" ON quality_scores
  FOR INSERT WITH CHECK (true);
```

The `WITH CHECK (true)` clause means no authentication or authorization is required. Any HTTP request to the Supabase REST API (even without a JWT token) can insert data into this table. The table contains fields like `user_id`, `relevance`, `accuracy`, `overall`, and `raw_response`.

## Proof of Concept

1. Send an INSERT request to the Supabase REST API for the `quality_scores` table without any Authorization header
2. The request will succeed because the RLS policy has `WITH CHECK (true)` — no authentication required
3. Verify the data was inserted by querying the table

```
# An unauthenticated POST to the Supabase REST API for quality_scores
# succeeds because the RLS policy has WITH CHECK (true)
import requests

SUPABASE_URL = "https://your-project.supabase.co"
ANON_KEY = "your-anon-key"

# No JWT token needed - just anon key (which is public)
response = requests.post(
    f"{SUPABASE_URL}/rest/v1/quality_scores",
    headers={
        "apikey": ANON_KEY,
        "Content-Type": "application/json"
        # NO Authorization header - unauthenticated
    },
    json={
        "user_id": "00000000-0000-0000-0000-000000000000",
        "relevance": 0.1,
        "accuracy": 0.1,
        "overall": 0.1
    }
)
print(f"Anonymous insert status: {response.status_code}")
print(f"Response: {response.text}")
```

## Code Analysis

**Location 1:** `supabase/migrations/00028_add_quality_scores.sql` (lines 34-35)
  Overly permissive INSERT policy
  ```
  CREATE POLICY "insert scores" ON quality_scores
  FOR INSERT WITH CHECK (true);
  ```

  **Suggested Fix:**
```diff
- CREATE POLICY "insert scores" ON quality_scores
-   FOR INSERT WITH CHECK (true);
+ CREATE POLICY "insert scores" ON quality_scores
+   FOR INSERT WITH CHECK (auth.role() = 'authenticated');
```

## Remediation

Replace the permissive policy in migration 00028 with one that requires authentication:

```sql
DROP POLICY IF EXISTS "insert scores" ON quality_scores;
CREATE POLICY "insert scores" ON quality_scores
  FOR INSERT WITH CHECK (auth.role() = 'authenticated');
```

Alternatively, if only admin users should be able to write scores:
```sql
DROP POLICY IF EXISTS "insert scores" ON quality_scores;
CREATE POLICY "insert scores" ON quality_scores
  FOR INSERT WITH CHECK (
    EXISTS (SELECT 1 FROM profiles WHERE id = auth.uid() AND role = 'admin')
  );
```

Apply this change by creating a new migration file (e.g., 00029_fix_quality_scores_rls.sql).
