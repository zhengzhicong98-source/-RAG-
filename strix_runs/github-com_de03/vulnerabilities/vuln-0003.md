# Overly Permissive RLS Policy - trace_logs INSERT Allows Anonymous Writes

**ID:** vuln-0003
**Severity:** MEDIUM
**Found:** 2026-07-12 09:58:11 UTC
**Target:** https://github.com/zhengzhicong98-source/legal-assistant
**Endpoint:** supabase/migrations/00021_add_trace.sql
**CWE:** CWE-862
**CVSS:** 5.3

## Description

The `trace_logs` table in migration 00021 has an INSERT RLS policy that allows any request (including unauthenticated) to insert rows. The policy `"users can insert trace" ON trace_logs FOR INSERT WITH CHECK (true)` requires no authentication or authorization.

## Impact

An unauthenticated attacker can inject arbitrary trace log entries into the trace_logs table, potentially filling the database with spam data, polluting the trace monitoring view in the admin dashboard, or injecting malicious JSONB metadata payloads.

## Technical Analysis

Migration `supabase/migrations/00021_add_trace.sql` at line 20 creates:

```sql
CREATE POLICY "users can insert trace" ON trace_logs
  FOR INSERT WITH CHECK (true);
```

The `WITH CHECK (true)` clause means anyone can insert records. The `metadata` field is JSONB, which could allow various injection attempts.

## Proof of Concept

1. Send an INSERT request to the Supabase REST API for the `trace_logs` table without any Authorization header
2. The request will succeed because the RLS policy allows all inserts
3. The attacker can inject arbitrary trace_id, span_name, and metadata values

```
# Anonymous insert into trace_logs succeeds because RLS policy is WITH CHECK (true)
import requests

SUPABASE_URL = "https://your-project.supabase.co"
ANON_KEY = "your-anon-key"

response = requests.post(
    f"{SUPABASE_URL}/rest/v1/trace_logs",
    headers={
        "apikey": ANON_KEY,
        "Content-Type": "application/json"
        # No Authorization header
    },
    json={
        "trace_id": "attacker_trace_001",
        "span_name": "spam",
        "start_time": "2026-01-01T00:00:00Z",
        "metadata": {"injected": True}
    }
)
print(f"Anonymous insert status: {response.status_code}")
```

## Code Analysis

**Location 1:** `supabase/migrations/00021_add_trace.sql` (lines 20-21)
  Overly permissive INSERT policy
  ```
  CREATE POLICY "users can insert trace" ON trace_logs
  FOR INSERT WITH CHECK (true);
  ```

  **Suggested Fix:**
```diff
- CREATE POLICY "users can insert trace" ON trace_logs
-   FOR INSERT WITH CHECK (true);
+ CREATE POLICY "users can insert trace" ON trace_logs
+   FOR INSERT WITH CHECK (auth.role() = 'authenticated');
```

## Remediation

Replace the permissive policy in migration 00021:

```sql
DROP POLICY IF EXISTS "users can insert trace" ON trace_logs;
CREATE POLICY "users can insert trace" ON trace_logs
  FOR INSERT WITH CHECK (auth.role() = 'authenticated');
```

Optionally bind user_id to the authenticated user:
```sql
CREATE POLICY "users can insert trace" ON trace_logs
  FOR INSERT WITH CHECK (
    auth.role() = 'authenticated' AND
    (user_id IS NULL OR user_id = auth.uid())
  );
```
