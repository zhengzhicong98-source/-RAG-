# Overly Permissive RLS Policy - ai_call_logs INSERT via NULL user_id

**ID:** vuln-0005
**Severity:** MEDIUM
**Found:** 2026-07-12 09:58:32 UTC
**Target:** https://github.com/zhengzhicong98-source/legal-assistant
**Endpoint:** supabase/migrations/00006_add_observability.sql
**Method:** POST
**CWE:** CWE-862
**CVSS:** 5.3

## Description

The ai_call_logs table INSERT RLS policy allows inserts with `user_id IS NULL`, meaning an unauthenticated (or any authenticated) request can insert log records without linking them to a user. This allows anonymous injection of AI call log entries.

## Impact

An attacker can inject fake AI call logs into the ai_call_logs table without authentication. These records would not be linked to any user, polluting observability data and potentially masking real issues.

## Technical Analysis

Migration `supabase/migrations/00006_add_observability.sql` at line 31:

```sql
CREATE POLICY "users can insert own logs" ON ai_call_logs
  FOR INSERT WITH CHECK (auth.uid() = user_id OR user_id IS NULL);
```

The `OR user_id IS NULL` clause allows anonymous/unauthenticated inserts into the ai_call_logs table.

## Proof of Concept

Send an INSERT request to the Supabase REST API for ai_call_logs with user_id set to null. The INSERT will succeed because the policy allows user_id IS NULL.

```
import requests

SUPABASE_URL = "https://your-project.supabase.co"
ANON_KEY = "your-anon-key"

# Insert without user_id (IS NULL) succeeds
response = requests.post(
    f"{SUPABASE_URL}/rest/v1/ai_call_logs",
    headers={"apikey": ANON_KEY, "Content-Type": "application/json"},
    json={
        "function_name": "faked",
        "model": "fake-model",
        "success": False
    }
)
print(f"Status: {response.status_code}")
```

## Code Analysis

**Location 1:** `supabase/migrations/00006_add_observability.sql` (lines 30-32)
  Overly permissive INSERT policy allows null user_id

  **Suggested Fix:**
```diff
- CREATE POLICY "users can insert own logs" ON ai_call_logs
-   FOR INSERT WITH CHECK (auth.uid() = user_id OR user_id IS NULL);
+ CREATE POLICY "users can insert own logs" ON ai_call_logs
+   FOR INSERT WITH CHECK (auth.uid() = user_id);
```

## Remediation

Modify the INSERT policy in migration 00006 to require user_id matching the authenticated user:

```sql
DROP POLICY IF EXISTS "users can insert own logs" ON ai_call_logs;
CREATE POLICY "users can insert own logs" ON ai_call_logs
  FOR INSERT WITH CHECK (auth.uid() = user_id);
```
